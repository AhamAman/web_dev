# Topic 6: HTTP & HTTPS Protocols — Roadmap Checklist

## Level 1 — Beginner: What HTTP actually is and why it exists

- [ ] **Understand the layer HTTP sits on.** TCP (Topic 5) only gives you a reliable stream of bytes between two machines — it has no idea what those bytes *mean*. HTTP is the agreement both sides make about how to format a "give me this resource" message and a "here's the resource" reply on top of that stream. Without this shared language, a browser and a server would just be exchanging meaningless bytes.
- [ ] **Grasp the request-response model.** Every HTTP exchange is a client sending exactly one request and a server sending exactly one response back. There's no server "pushing" data to you uninvited in plain HTTP — you always ask first.
- [ ] **Understand statelessness.** By default, HTTP has no memory of previous requests — each one is handled in total isolation. This is a deliberate first-principles design choice (it keeps servers simple and scalable), and it's the *reason* mechanisms like cookies and sessions had to be invented later to fake "memory" on top of a memoryless protocol.
- [ ] **Learn the anatomy of a request:** a method (GET, POST, etc.), a path (`/users/42`), a version (HTTP/1.1), a set of headers (metadata like `Host`, `User-Agent`, `Content-Type`), and an optional body.
- [ ] **Learn the anatomy of a response:** a status line (status code + short reason phrase), headers, and an optional body — usually the actual HTML, JSON, image, etc. you asked for.
- [ ] **Know the core request methods and what each *means*, not just its name:** `GET` (retrieve, shouldn't change anything on the server), `POST` (create/submit something new), `PUT` (replace a resource entirely), `DELETE` (remove a resource).
- [ ] **Know the three status codes you'll see constantly, and *why* each category exists:** `200 OK` (success), `404 Not Found` (the resource genuinely doesn't exist at that path), `500 Internal Server Error` (the server itself broke while handling a valid request — distinct from the client doing anything wrong).
- [ ] **Know, at a surface level, what HTTPS is:** the same HTTP, but wrapped in an encryption layer (TLS) so a third party watching the network traffic can't read or tamper with it.
- [ ] **Hands-on:** open your browser DevTools → Network tab, click any request, and actually read the raw request and response headers. Don't skip this — seeing real headers makes the abstract concept concrete fast.

## Level 2 — Intermediate: The mechanics underneath

- [ ] **Understand why HTTPS had to exist (first principles, not just "it's the secure one").** Plain HTTP fails at three separate problems simultaneously: *confidentiality* (anyone on the network path can read your data), *integrity* (anyone on the path can silently modify your data in transit), and *authentication* (you have no proof the server you're talking to is who it claims to be, rather than an impersonator). TLS exists specifically to solve all three at once.
- [ ] **Learn the full status code taxonomy by category, not just memorized numbers:** 1xx (informational, rare), 2xx (success), 3xx (redirection — the resource moved, follow this new location), 4xx (the *client* made a bad request — `401` means "I don't know who you are," `403` means "I know who you are and you're not allowed," `404` means "doesn't exist," `429` means "you're sending too many requests"), 5xx (the *server* failed, not your fault).
- [ ] **Understand idempotency and safety as properties, not trivia.** GET, PUT, and DELETE are idempotent (doing them twice has the same effect as once); POST is not (submitting the same form twice usually creates two records). This distinction is *why* browsers warn you before resubmitting a form on refresh.
- [ ] **Learn how cookies and sessions patch over statelessness.** The server hands the client a small token in a `Set-Cookie` header; the client echoes it back on every future request via the `Cookie` header, letting the server reconstruct "memory" it otherwise doesn't have.
- [ ] **Understand the difference between symmetric and asymmetric encryption, and why TLS uses both.** Asymmetric encryption (public/private key pairs) is computationally expensive but solves the "we've never met, how do we agree on a secret" problem. Symmetric encryption is fast but requires both sides to already share a secret. TLS uses asymmetric crypto *only* during the handshake to safely agree on a shared secret, then switches to fast symmetric encryption for all the actual data — the best of both, used for exactly what each is good at.
- [ ] **Get a conceptual (not yet expert) grasp of the TLS handshake:** client and server negotiate which crypto methods to use, the server proves its identity with a certificate, and both sides derive a shared symmetric key — all *before* a single byte of your actual HTTP request is sent.
- [ ] **Understand what a Certificate Authority (CA) actually solves.** The hard problem isn't encryption math — it's trust: how does your browser know the public key it just received really belongs to `example.com` and not an attacker pretending to be `example.com`? A CA is a third party your browser already trusts in advance (its certificate is pre-installed in your OS/browser), which vouches for the server by digitally signing its certificate. You're not trusting the server directly — you're trusting the CA's signature on the server's claim.
- [ ] **Hands-on:** click the padlock icon on any HTTPS site and inspect the certificate — look at who issued it, who it was issued to, and its expiry date.

## Level 3 — Expert: Performance, security, and real-world systems

- [ ] **Understand the chain of trust, not just "a CA signed it."** Browsers don't trust every CA directly — they trust a small set of root CAs, which sign intermediate CAs, which sign the actual server certificate. Verifying a certificate means walking this chain back to a root your browser already trusts. This indirection is *why* a compromised intermediate CA is such a serious event — it can undermine trust in every certificate it ever signed.
- [ ] **Know how TLS 1.3 improved on TLS 1.2's handshake.** TLS 1.2 typically needed 2 round trips before encrypted application data could flow; TLS 1.3 cut this to 1 round trip by negotiating fewer options up front, and added 0-RTT resumption for servers you've connected to recently — directly attacking the same latency problem QUIC attacks at the transport layer.
- [ ] **Understand HTTP/1.1 vs HTTP/2 vs HTTP/3 at the level of *what problem each one fixed*.** HTTP/1.1 sends requests over a connection mostly one-at-a-time (head-of-line blocking). HTTP/2 introduced multiplexing — many requests share one TCP connection simultaneously — plus header compression (HPACK) to cut repeated overhead. HTTP/3 goes further by running over QUIC (UDP-based) instead of TCP, eliminating head-of-line blocking *at the transport level itself*, not just the application level.
- [ ] **Understand HSTS (HTTP Strict Transport Security).** A server header that tells the browser "never attempt plain HTTP with me again, even if a user types it" — closing the window where a man-in-the-middle could intercept the very first plaintext request before HTTPS kicks in.
- [ ] **Understand certificate revocation (OCSP/CRL).** A certificate can be issued correctly and later need to be invalidated (e.g., the private key leaked). OCSP lets a browser ask "is this certificate still valid?" in near real time rather than trusting it blindly until expiry.
- [ ] **Understand CORS (Cross-Origin Resource Sharing) as a security boundary enforced via headers**, including why browsers send an automatic `OPTIONS` preflight request before certain cross-origin calls — the browser is asking the *server* for permission on the client's behalf before sending the real request.
- [ ] **Understand HTTP caching mechanics in depth:** `Cache-Control` directives, `ETag`/`If-None-Match` for validation, `Last-Modified`/`If-Modified-Since` as the older equivalent, and how CDNs use these same headers to cache content at edge locations close to users.
- [ ] **Know the common security-hardening response headers** beyond HSTS: `Content-Security-Policy` (restricts what sources scripts/styles/images can load from), `X-Frame-Options` (prevents clickjacking via iframes) — worth a deeper dedicated topic later, but good to recognize now.
- [ ] **Hands-on:** run `curl -v https://example.com` and read the TLS handshake details it prints; run `openssl s_client -connect example.com:443` to inspect the full certificate chain manually; try sending the same request with and without a cache-busting header and watch the timing difference in DevTools.

## Quick self-check before moving on

You should be able to explain, without looking anything up: why HTTP needed to exist on top of TCP at all, why statelessness was a deliberate design choice rather than an oversight, why TLS needs both symmetric and asymmetric encryption instead of just one, and what specific problem a Certificate Authority solves that encryption alone cannot.
