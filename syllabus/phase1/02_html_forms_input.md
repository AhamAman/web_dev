# Topic 2: HTML Forms and Inputs — User Interaction (Roadmap Checklist)

## Level 1 — Beginner: What forms actually are and why they exist

- [ ] **Understand the problem forms solve.** Plain HTML is fundamentally read-only — headings, paragraphs, and links let a browser *display* content, but none of it lets a user *send information back*. Forms are the original, foundational mechanism for collecting structured input from a user and transmitting it somewhere — every "interactive" thing on the web, long before JavaScript existed, traces back to this one primitive.
- [ ] **Learn the role of `<form>` as a container with instructions attached.** The `<form>` element wraps a group of related inputs and carries two critical attributes: `action` (where the data should be sent) and `method` (how it should be sent). Without a form wrapper, a browser has no way to know which inputs belong together as one submission.
- [ ] **Learn that `<input type="...">` is one element with many personalities.** The `type` attribute doesn't just change appearance — it changes the actual behavior: `type="text"` is free-form, `type="email"` triggers basic format checking and an `@`-friendly mobile keyboard, `type="number"` gives a spinner and numeric keyboard, `type="password"` masks the characters visually as you type.
- [ ] **Learn `<textarea>` vs `<input type="text">`.** A single-line `<input>` is for short values (a name, an email); `<textarea>` is for multi-line free text (a comment, a message) and is a genuinely different element rather than just a styled-bigger input.
- [ ] **Learn `<select>` and `<option>` for fixed-choice input.** When the valid answers are a known, limited set (a country, a size), a dropdown constrains the user to valid choices rather than letting them type anything.
- [ ] **Learn the `<button>` types and why they're not interchangeable.** `type="submit"` triggers form submission, `type="reset"` clears the form back to defaults, `type="button"` does nothing on its own and exists purely as a hook for custom JavaScript behavior.
- [ ] **Understand that `name`, not `id`, is what gets sent to the server.** Each input's `name` attribute becomes the key in the submitted data (`name=John`); `id` is only for linking labels and CSS/JS targeting and is never transmitted as form data. Forgetting `name` is the single most common reason a "working-looking" form silently submits nothing useful.
- [ ] **Reinforce label association from Topic 1, specifically for forms.** Every input should have an associated `<label>` (via a matching `for`/`id` pair, or by wrapping the input inside the label) — this isn't cosmetic, it's what lets clicking the label text focus the input, and what lets a screen reader announce what the field is for.
- [ ] **Hands-on:** build a simple contact form with a couple of text inputs and a submit button, with no JavaScript at all, submit it, and watch what happens to the URL — this sets up the next section perfectly.

## Level 2 — Intermediate: Submission mechanics, validation, and accessible structure

- [ ] **Understand the real difference between GET and POST, from first principles, not from memorized rule.** A GET form submission packs all the field data into the URL as a query string (`?name=John&age=30`) — visible in the address bar, browser history, and server access logs, and subject to URL length limits. A POST submission sends the data inside the request body instead — invisible in the URL, with no practical length restriction. The deeper principle: GET is semantically "retrieve/search," so it suits things like search filters where the URL *should* be shareable and bookmarkable; POST is semantically "submit/change," so it suits anything that creates, updates, or contains sensitive data, where you don't want it logged in plaintext in a URL.
- [ ] **Understand what actually happens when a form submits, mechanically.** The browser walks every named form control inside the `<form>`, collects their current values, encodes them into a single payload, and either appends that payload to the URL (GET) or places it in the request body (POST) — then performs a full page navigation to whatever `action` points to, unless JavaScript intercepts the event first.
- [ ] **Learn the `enctype` attribute and why it has to change for file uploads.** The default encoding, `application/x-www-form-urlencoded`, can't represent binary file data — uploading a file requires switching to `multipart/form-data`, which packages each field (including file bytes) as a separate labeled chunk. This is a real, easy-to-forget gotcha: a file input inside a form with the default `enctype` will silently fail to upload the actual file content.
- [ ] **Learn how radio buttons are grouped, and why it isn't magic.** Multiple `<input type="radio">` elements sharing the exact same `name` attribute become mutually exclusive automatically — the browser enforces "only one selected" purely because they're recognized as the same logical group via that shared name, not because of any special radio-specific wiring.
- [ ] **Learn the native validation attributes and what they actually enforce:** `required` (can't submit empty), `min`/`max` (numeric or date bounds), `minlength`/`maxlength` (text length bounds), `pattern` (a regular expression the value must match). These are part of the browser's built-in Constraint Validation API — the browser blocks submission and shows a native error bubble automatically, with zero JavaScript required.
- [ ] **Internalize the most important rule about validation: client-side validation is a UX feature, not a security feature.** It runs entirely inside the user's own browser, which means it can be trivially bypassed (disabling JavaScript, editing DevTools, or just sending a raw HTTP request with curl). Anything the server actually depends on for correctness or security *must* be re-validated server-side — client-side checks only exist to give the user instant feedback before a round trip is even attempted.
- [ ] **Learn proper grouping and labeling for accessible forms:** `<fieldset>` and `<legend>` group a set of related controls (like a set of radio buttons) under one shared description, and `placeholder` text is not a substitute for a real `<label>` — placeholder text disappears the moment a user starts typing, leaving no persistent field description for anyone who looks away and back, or who relies on assistive technology.

## Level 3 — Expert: Programmatic control, security, and real-world rigor

- [ ] **Learn the Constraint Validation API in JavaScript** — `element.checkValidity()`, `element.setCustomValidity("message")`, and the `ValidityState` object let you layer custom validation logic (like "passwords must match") on top of native HTML validation, rather than abandoning native validation entirely and reinventing it all in JS.
- [ ] **Apply progressive enhancement specifically to forms.** A form should work via plain native submission with zero JavaScript as the baseline — JavaScript should *enhance* that baseline (intercepting submit to send the data via `fetch` instead, for a smoother single-page experience) rather than being the only path that works at all. This directly extends the progressive enhancement principle from HTML Basics.
- [ ] **Understand why GET should never be used for state-changing actions, security-wise, beyond just "it's visible."** Browsers and crawlers sometimes prefetch or revisit GET URLs automatically (assuming GET is always safe/read-only by HTTP convention), so a "delete account" action wired to a GET request can be triggered completely accidentally. This is also tied to CSRF risk: a malicious page can trivially trigger a GET request against your site just by embedding an image tag pointing at it, with no special exploit needed.
- [ ] **Always assume an attacker can submit anything, regardless of your HTML.** Since validation attributes and even entire form fields can be bypassed by sending a raw request directly to your endpoint, server-side validation must independently re-check everything (required-ness, ranges, formats) as if the HTML form never existed at all.
- [ ] **Understand file upload specifics beyond the `enctype` basics:** the `accept` attribute on a file input is only a *hint* to the browser's file picker UI (e.g., "show mostly images") — it does not restrict or enforce what file type actually gets uploaded, so server-side file type/size validation is non-negotiable.
- [ ] **Know how `autocomplete` attribute values (like `autocomplete="email"` or `autocomplete="new-password"`) hook into browser autofill heuristics**, since getting these right meaningfully improves real-world completion rates and getting them wrong (or omitting them) is a common, easily fixed UX papercut.
- [ ] **Learn the `FormData` API for JavaScript-driven submissions.** When intercepting a form submit to send data via `fetch` instead of letting the browser navigate, `new FormData(formElement)` captures all named fields (including files) in the same structure a native submission would have used — letting you preserve the same `multipart/form-data`-style encoding for an AJAX request.
- [ ] **Recognize where this is heading in modern frameworks:** the distinction between "uncontrolled" inputs (the DOM holds the current value, as in plain HTML) and "controlled" inputs (a JavaScript framework like React holds the value in state and the DOM just reflects it) is a direct evolution of everything covered here — worth keeping in mind once you reach component-based frameworks.
- [ ] **Hands-on:** build a small registration form using several different input types plus at least one validation attribute; inspect the Network tab to confirm the practical difference between a GET submission (data in the URL) and a POST submission (data in the request body); then deliberately bypass your own client-side validation using curl or DevTools to prove to yourself why server-side re-validation is mandatory, not optional.

## Quick self-check before moving on

You should be able to explain, without looking anything up: why `name` (not `id`) determines what gets submitted, why GET and POST aren't just "two configuration options" but reflect different semantic intents, why client-side validation can never be trusted as a security boundary, and why a file upload silently fails without the correct `enctype`.
