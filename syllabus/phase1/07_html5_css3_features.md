# Topic 7: HTML5 & CSS3 Features (Roadmap Checklist)

*A few items in your syllabus here (semantic elements, certain input types, transitions) were already covered in depth in earlier topics — this guide links back to those rather than repeating them, and focuses on what's genuinely new: the newer input types, native CSS3 visual effects, and the cross-browser compatibility story behind all of it.*

## Level 1 — Beginner: What "HTML5/CSS3" actually meant as an upgrade

- [ ] **Understand that "HTML5" wasn't just a batch of new tags — it was a philosophy shift.** Earlier HTML standardization efforts pushed toward strict, XML-like document markup. HTML5 deliberately leaned the other way: it embraced how the web was *actually* being used — as a platform for applications, video, audio, and interactive UI — and standardized patterns developers had already converged on informally, rather than enforcing theoretical purity.
- [ ] **Recognize semantic elements (`header`, `footer`, `main`, `article`, `section`) as a direct example of that philosophy** — covered in full depth in the HTML Basics topic, so revisit that if any of these feel unfamiliar. The short version of *why* they matter here: before HTML5, every project independently invented `<div class="header">`, `<div class="footer">`, etc. — visually fine, but meaningless to a browser or screen reader. HTML5 turned the *pattern* developers had already agreed on into actual semantics with real, built-in meaning.
- [ ] **Learn the genuinely new HTML5 input types your syllabus calls out:** `date` (gives a native date-picker UI instead of free-text typing), `range` (gives a slider for picking a numeric value within a min/max bound), and `color` (gives a native color-picker UI that returns a hex value). `email` and `tel` were already covered in the HTML Forms topic — same underlying idea, just simpler validation/keyboard hints rather than a whole custom picker UI.
- [ ] **Understand the first-principles reason these input types matter: they offload UI building and basic validation work the browser can do natively.** Before these existed, every project that needed a date picker or color picker had to build (or bundle a JS library for) that whole interactive widget from scratch — now the browser engine ships it for free, consistent with the OS's native picker conventions.
- [ ] **Learn the new CSS3 visual properties at a surface level:** `border-radius` (rounded corners), `box-shadow` (drop shadows), and gradient backgrounds (`linear-gradient()`, `radial-gradient()`) — all achievable with pure CSS declarations, no image files involved. `transition` itself was already covered in depth in the Advanced CSS Styling topic.
- [ ] **Understand the concrete problem these CSS3 properties solved.** Before CSS3, rounded corners, shadows, and gradients required actual sliced image assets (extra HTTP requests, fixed pixel dimensions, can't be resized or recolored without regenerating the image file). Moving these into pure CSS removed that entire image-asset workflow for purely decorative effects — they became resizable, themeable, and free of any extra network request.
- [ ] **Hands-on:** build a small form using `date`, `range`, and `color` inputs with zero custom JavaScript, and notice how much native, free interactive UI you get; then style a plain `<div>` with `border-radius`, `box-shadow`, and a `linear-gradient` background to see the same visual effects you'd previously have needed an image for.

## Level 2 — Intermediate: Cross-browser reality and the syntax details that matter

- [ ] **Understand the cross-browser compatibility problem at its root.** Different browser engines (historically Trident/IE, WebKit, Gecko, and today mainly Blink, WebKit, and Gecko) have never implemented every new CSS/HTML feature at exactly the same time, with exactly the same behavior. A feature can be fully supported in one browser and missing or buggy in another for months or years — which is precisely why "does this actually work everywhere" has to be a deliberate check, not an assumption.
- [ ] **Learn what vendor prefixes were, and why they existed.** Properties like `-webkit-border-radius` or `-moz-box-shadow` let a browser engine ship an experimental, not-yet-finalized implementation of an upcoming spec feature without claiming full, stable, unprefixed conformance. They were meant to be a temporary bridge while a spec stabilized — most CSS3 properties covered here have long since dropped their prefix requirement in modern browsers, but you'll still see them in older code and tutorials.
- [ ] **Make checking actual support data (caniuse.com, MDN's compatibility tables) a habit, not an afterthought**, before relying on any newer feature in production — "I think this works everywhere now" is a guess; a compatibility table is a fact.
- [ ] **Learn `border-radius` and `box-shadow` syntax in real depth, not just the single-value case.** `border-radius` accepts up to four values to give each corner a different radius (or elliptical corners with a slash syntax); `box-shadow` accepts multiple comma-separated shadow definitions layered on top of each other, plus an `inset` keyword to draw the shadow *inside* the element's edge instead of outside it.
- [ ] **Learn gradient syntax beyond "two colors."** `linear-gradient()` takes a direction (or angle) followed by any number of color stops, optionally with explicit position percentages for each stop — letting you build complex multi-color transitions, not just a simple two-color fade. `radial-gradient()` does the same thing but radiating outward from a center point instead of along a line.
- [ ] **Notice a genuine, concrete cross-browser example sitting right inside this topic: the `date` input.** Every browser implements its own native date-picker UI for `type="date"`, and those UIs look meaningfully different from each other (and the value format shown to the user can vary even though the underlying submitted value is standardized) — a perfect small case study for why "native doesn't mean visually identical everywhere."
- [ ] **Learn that `type="range"` only gives you the slider — it doesn't display the current numeric value on its own.** Most real implementations pair it with a small bit of JavaScript that listens for the input event and writes the live value into a nearby element, since the native slider UI alone shows no number.

## Level 3 — Expert: Feature detection, tooling, performance, and accessibility payoff

- [ ] **Learn feature detection as the correct modern replacement for old-style browser sniffing.** Sniffing the user-agent string to guess what a browser supports is unreliable (strings are spoofable and inconsistent) and doesn't scale as new browsers and versions appear constantly. Feature detection instead asks the browser directly whether it actually supports a given capability — checking for the real thing you need, not a proxy for it.
- [ ] **Learn the `@supports` CSS at-rule as feature detection built directly into CSS itself.** `@supports (backdrop-filter: blur(10px)) { ... }` lets you write one block of CSS for browsers that support a feature and a separate fallback outside it for those that don't — querying actual capability rather than guessing from a browser name or version number.
- [ ] **Understand graceful degradation as the cascade quietly doing its job, even without `@supports`.** If a browser doesn't recognize a CSS value (an old browser seeing `background: linear-gradient(...)`, for instance), it simply ignores that one declaration rather than erroring out — which means declaring a plain `background-color` fallback *before* a gradient declaration in the same rule lets unsupported browsers fall back automatically, purely through normal cascade order, with no `@supports` query needed at all.
- [ ] **Know what polyfills are and where they fit.** A polyfill is a JavaScript library that detects a missing native feature and reimplements its behavior manually — for instance, detecting that `type="date"` silently degraded to a plain text input in an unsupported browser, and injecting a custom JS-built date picker to restore the missing functionality. This is the practical, code-level version of the same graceful-degradation principle: an unsupported input type doesn't break the form, it just quietly becomes less capable, and a polyfill can choose to patch that gap.
- [ ] **Know that modern build tooling automates vendor prefixing entirely.** Tools like Autoprefixer generate the correct prefixed CSS automatically based on a declared target browser support list, which is why hand-writing `-webkit-`/`-moz-` prefixes yourself is now mostly a thing of the past in any project using a real build pipeline — a direct extension of the build-tooling concepts introduced in the CSS Frameworks topic.
- [ ] **Connect CSS3 visual effects back to the performance principles from Advanced CSS Styling.** Gradients and shadows are generally cheap to *render once*, but animating `box-shadow` directly (rather than via `transform`/`opacity` tricks) forces repaint on every frame, and a large blur radius on `box-shadow` is measurably more expensive to compute than a sharp one — the same compositor-vs-paint cost distinction from before applies just as much to these "new" properties as to anything else.
- [ ] **Know that HTML5's "document outline algorithm" (where nested `section`/`article` elements were meant to imply an automatic heading outline) was specified but never properly implemented consistently by browsers, and has effectively been abandoned** — in practice, your actual heading hierarchy (`h1`–`h6`, covered in HTML Basics) is what assistive technology and tooling rely on, not nesting depth of sectioning elements.
- [ ] **Recognize the accessibility argument for preferring native input types over custom-built equivalents whenever possible.** Native `date`, `range`, and `color` inputs generally ship with solid built-in keyboard operability and correct semantics out of the box, where a fully custom JavaScript-built date picker or slider has to deliberately reimplement all of that correctly — "use the native element unless you have a real design reason not to" is a genuinely strong default, not just a lazy shortcut.
- [ ] **Hands-on:** render the same `<input type="date">` in two different browsers side by side and compare the native picker UI directly; write a CSS rule with a plain `background-color` fallback declared before a `linear-gradient`, then use DevTools to disable the gradient support (or just trust the cascade order) to confirm the fallback logic; check caniuse.com for a CSS property you assume is "obviously fine everywhere" and see exactly how recently that assumption became true.

## Quick self-check before moving on

You should be able to explain, without looking anything up: why semantic HTML5 elements were really just standardizing patterns developers already used, why a `date` input behaving differently across browsers is a textbook cross-browser compatibility issue rather than a bug, how cascade order alone can provide a CSS fallback without any `@supports` query, and why animating `box-shadow` directly is more expensive than animating `transform`/`opacity`.
