# Topic 1: HTML Basics — The Web's Skeleton (Roadmap Checklist)

## Level 1 — Beginner: What HTML actually is and why it exists

- [ ] **Understand the problem HTML solves before learning any tags.** A raw block of text has no structure a program can understand — there's no way to tell a browser "this part is a heading, this part is a link, this part is a list" just from plain characters. HTML is a labeling system: it wraps content in tags that declare *what role each piece plays*, so a browser (or a screen reader, or a search engine) can interpret meaning, not just display a wall of text.
- [ ] **Learn the difference between a tag and an element, precisely.** A tag is the markup syntax itself — `<p>` is an opening tag, `</p>` is a closing tag. An *element* is the tag pair plus everything between them: `<p>Hello</p>` is the element. Some elements are self-closing/"void" because they never wrap content, like `<img>` or `<br>` — there's nothing to put inside an image tag, so it doesn't need a closing tag.
- [ ] **Understand nesting as a tree, not just indentation.** Elements live inside other elements, forming a parent-child hierarchy — this is literally called the DOM (Document Object Model) once the browser parses it. Thinking of your page as a tree from day one will save you confusion later when CSS and JavaScript both rely on this same tree structure.
- [ ] **Understand attributes as configuration, not content.** An attribute is a key-value pair inside the opening tag that adds information *about* the element without being part of its visible content — `<a href="https://example.com">` tells the browser where the link goes; `href` itself never appears on the page.
- [ ] **Learn the page skeleton and what each part is *for*:** `<!DOCTYPE html>` tells the browser to render in modern standards mode rather than legacy "quirks mode." `<html>` is the root of the whole document tree. `<head>` holds metadata never shown directly on the page (title, character encoding, links to stylesheets). `<body>` holds everything actually visible to a user. `<header>` and `<footer>` mark the introductory and closing regions of a page (or of a section within it).
- [ ] **Learn the core text elements and their actual purpose, not just their look:** `<h1>` through `<h6>` express a hierarchy of importance (not just "bigger text" — a heading level is a semantic claim about document structure). `<p>` is a paragraph of text. `<a>` creates a hyperlink via its `href` attribute. `<ul>` is an unordered (bulleted) list, `<ol>` is an ordered (numbered) list, and `<li>` is each individual item inside either one.
- [ ] **Hands-on:** write a plain HTML file from scratch with no CSS at all, open it directly in a browser, and notice how much structure (headings, paragraphs, lists, links) is already visually distinguishable purely from semantic tags — that's the point of HTML existing as a separate layer from styling.

## Level 2 — Intermediate: Semantics, structure, and the DOM

- [ ] **Understand *why* semantic HTML matters, concretely, not just as a buzzword.** Three real consequences: accessibility (screen readers use semantic landmarks like `<nav>` or `<main>` to let blind users jump directly to relevant regions instead of reading the whole page linearly), SEO (search engines weigh heading structure and semantic regions when deciding what a page is actually about), and maintainability (another developer reading `<article>` instantly understands intent, where a generic `<div>` tells them nothing).
- [ ] **Learn the precise difference between `<section>`, `<article>`, and `<div>`.** `<article>` is for self-contained content that would still make sense if pulled out and syndicated elsewhere on its own — a blog post, a comment, a news story. `<section>` groups content thematically within a document (e.g., a "Reviews" section of a product page) without claiming it's independently reusable. `<div>` carries zero semantic meaning at all — it's a pure structural/styling hook, used only when no semantic element actually fits the content's role.
- [ ] **Learn the purpose of `<nav>`, `<main>`, and `<aside>`.** `<nav>` wraps a group of navigation links (not every link on the page — just navigational groups like a menu). `<main>` wraps the primary, unique content of the page, and should appear exactly once. `<aside>` wraps content that's tangentially related to the main content but not part of its core flow — sidebars, pull quotes, related-links widgets.
- [ ] **Understand that the browser turns your HTML into a live object tree (the DOM), not just a picture.** This matters because JavaScript doesn't manipulate "the HTML text" — it manipulates this tree object directly, and CSS selectors target nodes in this same tree. The HTML you write is really the *initial instructions* for building that tree, not the final, unchangeable output.
- [ ] **Learn the block vs. inline distinction and why it exists.** Block-level elements (like `<p>`, `<div>`, `<section>`) start on a new line and take up the full available width by default; inline elements (like `<a>`, `<span>`) flow within a line of text alongside other content. This default behavior comes from each element's role: a paragraph is meant to be its own visual block, a link is meant to sit inside a sentence.
- [ ] **Learn the global attributes that apply almost everywhere:** `id` (a unique identifier for one specific element), `class` (a reusable label shared across many elements, mainly for CSS/JS targeting), `title` (a tooltip), and `data-*` attributes (custom data a developer attaches for JavaScript to read later, with no built-in meaning to the browser itself).
- [ ] **Learn the basics of accessible markup:** the `alt` attribute on `<img>` describes the image for anyone who can't see it (screen reader users, or anyone whose image failed to load), and form `<label>` elements should be explicitly associated with their inputs so assistive technology can announce what each field is for.

## Level 3 — Expert: Rendering pipeline, performance, and real-world rigor

- [ ] **Understand where ARIA fits relative to semantic HTML.** The first rule of ARIA is: don't use it if a native semantic element already does the job — `<nav>` is always better than `<div role="navigation">`. ARIA roles and attributes exist to fill gaps where no native HTML element expresses the needed meaning (complex custom widgets like tab panels or comboboxes), not to replace semantics that already exist.
- [ ] **Understand the rendering pipeline your HTML feeds into:** the browser parses HTML into the DOM tree, parses CSS into a separate CSSOM tree, combines both into a render tree (which excludes anything set to `display: none`), computes layout (geometry/positions), and finally paints pixels. Your HTML structure is the very first input to this entire chain — a bloated or deeply nested DOM has real, measurable performance costs downstream in layout and paint.
- [ ] **Understand why the DOCTYPE declaration isn't just boilerplate.** Without a correct `<!DOCTYPE html>`, browsers fall back to "quirks mode," which recreates old, inconsistent rendering bugs from 1990s browsers for backward compatibility — a single missing line can silently change how every other element on your page is measured and positioned.
- [ ] **Understand progressive enhancement as a design philosophy, not a checklist item.** A well-built page should remain readable and navigable with just HTML — before any CSS or JavaScript loads, or if either fails entirely. This is precisely why semantic structure matters more than visual styling: structure is what survives when everything else is stripped away.
- [ ] **Know how heading hierarchy and document structure feed into both accessibility tooling and SEO ranking signals** — screen readers can generate a "table of contents" for a page purely from heading levels, and search engines use the same structure to judge topical relevance, so skipping heading levels for visual reasons (using `<h3>` because it "looks right" when no `<h2>` exists) actively breaks both.
- [ ] **Know the existence of custom elements / Web Components** as the mechanism for defining your own semantically meaningful tags beyond the built-in HTML vocabulary — worth a dedicated deep dive once you're comfortable with the basics, since it ties directly back into "semantics over generic divs" at a more advanced level.
- [ ] **Know that the `lang` attribute on `<html>` isn't cosmetic** — it tells screen readers which pronunciation rules to use and tells browsers/translation tools/search engines what language the content is actually in, affecting both accessibility and SEO simultaneously.
- [ ] **Hands-on:** open DevTools' Elements panel and explore the live DOM tree of a real website (not the page source, the parsed tree); run a Lighthouse accessibility audit on a page you've built and see exactly what semantic gaps it flags; deliberately remove a DOCTYPE from a test page and observe what visibly breaks.

## Quick self-check before moving on

You should be able to explain, without looking anything up: why a `<div>`-only page is technically valid but a structural failure, what concrete difference `<article>` vs `<section>` vs `<div>` makes to a screen reader user, why heading levels are a semantic hierarchy rather than a font-size shortcut, and why the DOM is described as a "live tree" rather than just "the HTML you wrote."
